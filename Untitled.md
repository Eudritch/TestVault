Here’s how I’d answer your stealth-first questions **if the goal is “users don’t feel like they’re deciding”** while keeping onboarding clean, professional, and fast.

---

## 1) First 30 seconds: What should users do first without thinking?

**Pick: A) Create something immediately.**  
Drop them into a real page with the cursor already active (a “Quick Note” / “Inbox” page). Creating one small thing is the fastest _aha_ and doesn’t feel like setup—just usage. Browsing/importing can be offered _after_ the first edit as optional nudges.

## 2) Hidden setup level: How much setup should happen silently in background?

**Pick: A) Most of it.**  
Generate defaults (pages, nav structure, directory) instantly so the app feels “ready.” The user should experience outcomes, not configuration. Anything that matters later should be reversible and discoverable in Settings.

## 3) Personalization style: How should we personalize?

**Pick: B) 1 quick preference question + behavior** _(even though A is tempting)._  
Pure behavior-first is clean, but it can take too long to infer intent and risks “blank-start confusion.” A single lightweight “Make this feel more like: Notes / Tasks / Calendar” chip row gives immediate direction, then behavior refines silently.

## 4) Decision visibility: Should users see “steps/progress”?

**Pick: A) No explicit steps.**  
Step indicators trigger “task mode.” If you need progress internally, show it as **soft state changes**: CTA becomes clearer, workspace becomes more “settled,” prompts disappear as they act.

## 5) Default strategy: How aggressive should smart defaults be?

**Pick: A) Strong defaults, easy edit later.**  
Strong defaults reduce choice fatigue and accelerate time-to-value. The key is making edits feel lightweight (rename, reorder, hide) and adding a “Reset to recommended” escape hatch. Users feel safe when nothing is permanent.

## 6) Desktop/tablet entry: What should the first screen feel like?

**Pick: A) Real workspace with subtle guide overlays.**  
Desktop should open straight into a live workspace with a quiet overlay that disappears on first interaction. The sidebar can be visible immediately because desktop supports peripheral context. Avoid a “Welcome screen”—it screams onboarding.

## 7) Mobile entry: What should mobile feel like?

**Pick: A) Swipeable mini-actions inside real app context.**  
Mobile should still start in the real page, but personalization should be tiny swipeable cards/chips (not deep structure editing). Keep it thumb-friendly, single-surface, and momentum-based. Save complex navigation shaping for later or presets.

## 8) Guidance tone: How should microcopy feel?

**Pick: A) Calm, clean, professional.**  
One sentence max, never instructional paragraphs. Use language that frames changes as reversible tuning (“Adjust anytime”) rather than setup. Microcopy should appear _only after_ interaction, then fade.

## 9) Motion style: How much animation?

**Pick: A) Subtle momentum cues only.**  
Use motion to clarify cause-and-effect (sidebar settles, items reorder smoothly, overlays fade), not to entertain. This reads premium and professional. Offer “Reduce motion” support and avoid bouncy springs.

## 10) Reversibility: Can every auto-decision be changed later?

**Pick: A) Yes, instantly.**  
Reversibility is what lets you hide decisions safely. Give users quick controls: “Switch navigation style,” “Add/remove starter pages,” “Reset workspace.” If it’s reversible, defaults can be stronger without feeling risky.

## 11) Trust-critical moments: What must be explicit?

**Pick: A) Privacy/data/storage only.**  
Explicit consent for sync, imports, integrations, analytics beyond necessary, billing—always. Keep everything else implied and reversible. If you ask too much up front, it becomes a questionnaire.

## 12) Success metric: What defines “onboarding worked”?

**Pick: A) First meaningful action completed fast.**  
Track: time to first edit, first page created, first organization action (rename/move), and whether they return (D1). Completion rate is less important because the best onboarding is often “skipped” without harm.

---

### Minimal stealth flow (what this implies)

```
Open app → Cursor is active in “Quick Note”
          → user types (aha)
          → optional chips: “Make it feel more like…”
          → optional preset: “Navigation style”
          → consent only when data is involved
```
